E1 Cache Concepts – JDB Cache – Part 2

In last article (part 1), have shared about the concept of JDB cache and its features. In this article, we will see how we can implement the JDB cache and use it efficiently during our performance issues.

How it works

When a table is configured to be a part of JDB cache then , once a record is fetched from the table, it gets stored in cache also and then further Selects of the same record are fetched from cache rather than the database.

Basic Rules

  1. An entry should be there in F98613 for the table which is a list of cache tables.
  2. Table must be accessed using full primary key.
  3. Table must be accesses through API ‘JDB_FetchKeyed’ in C Business Function or using ‘Fetch Single’ table I/O operation 1. in JDE event rules.
  4. APIs like JDB_SetSelection or JDB_SetSequencing must not be called when using the JDB cache.
  5. There is no synchronization / replication across processes.

Limitations

  1. Tables must be static.
  2. Same records must be fetched repeatedly.
  3. Caching tables which are dynamic will cause applications to break.
  4. Queries which return zero row, will not be cached.
  5. Caching table having millions of records is not suggested.

Benefits

  1. An automated table caching mechanism feature in EnterpriseOne.
  2. Suitable for table which remain constant and static.
  3. It can help in increasing performance of the process.
  4. Redundant Select statements on same primary key can be eliminated by using JDB cache.
